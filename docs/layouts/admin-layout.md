# SubHub 后台共享布局规范（Admin Layout）

## 1. 文档定位

本规范用于定义 SubHub 后台产品跨页面共享的布局骨架、区域组织规则与响应式布局行为。

它不是单页 page spec，也不替代全局设计系统文档；其职责是为页面规范、设计稿、前端实现与 UI 评审提供统一的布局基线，降低跨页面结构漂移。

### 1.1 与其他文档的边界

- `DESIGN.md`：负责全局设计系统、语义色彩、组件方向与通用交互原则。
- `docs/layouts/admin-layout.md`：负责后台共享布局规则、页面骨架类型与响应式外壳行为。
- `docs/pages/*.md`：负责单页面特例、模块顺序、页面级交互细节与断点例外。

### 1.2 使用原则

- 当页面未声明特例时，默认遵守本规范。
- 当页面有明确业务例外时，必须在对应 `docs/pages/*.md` 显式记录，不得隐式偏离。
- 单页偶发结构不得直接提升为全局布局规则。

## 2. 适用范围

本规范适用于 SubHub 后台 Shell 内的大多数控制台页面，典型包括但不限于：

- dashboard
- providers
- provider-detail
- users
- access-control
- settings
- api-keys

### 2.1 非完全适用页面

以下页面不完全适用本规范，需以页面规范为准：

- 登录页、邀请接受页、重置密码页等认证入口页面
- 首次引导或安装向导页
- 明确脱离后台 Shell 的全屏流程页

## 3. 后台 Shell 结构

后台共享 Shell 由以下区域组成：

1. Sidebar（主导航区）
2. Topbar / Page Header（页面头部区）
3. Main Content Area（主内容区）
4. Secondary Panel（可选次级面板区）

### 3.1 Sidebar

- 角色：承载一级导航、工作区切换、全局状态入口。
- 优先级：高于内容区视觉变化，低于全局阻断反馈（例如系统级错误）。
- 布局原则：
  - 宽屏下保持稳定可预期宽度，不随页面内容波动。
  - 导航层级与排序跨页面保持一致，不因单页临时需求重排。
  - 支持折叠态，但折叠后仍需保留可识别导航语义（图标 + 提示）。

### 3.2 Topbar / Page Header

- 角色：承载页面标题、页面级摘要、关键主操作、范围切换与上下文状态。
- 优先级：高于页面内局部区块，作为“当前页面语义入口”。
- 布局原则：
  - 标题与主操作位于首屏可见范围。
  - 页面级筛选不应与全局导航混放。
  - Header 不承担大面积分析内容，只承载“进入页面后立即要做的判断与操作”。

### 3.3 Main Content Area

- 角色：承载页面主体模块（表格、卡片、详情、表单、日志、状态块）。
- 优先级：页面信息密度最高区域，遵循明确的信息层级与可扫描结构。
- 布局原则：
  - 优先模块化分区，避免自由流拼贴式排版。
  - 区块间距、分组逻辑在同类页面保持一致。
  - 允许按页面骨架切换单栏、双栏或混合，但切换逻辑须可预期。

### 3.4 Secondary Panel（可选）

- 角色：承载辅助信息、上下文检查、事件时间线、联动详情或轻量编辑。
- 优先级：低于主任务流，不得遮蔽主操作路径。
- 布局原则：
  - 仅在确有“主内容 + 辅助上下文”并行阅读需求时启用。
  - 次级面板应可收起或替代为抽屉，不应强制永久占据主内容宽度。
  - 不应用作临时堆放非结构化信息的容器。

## 4. 页面骨架类型

本节定义跨页面可复用的骨架类型。页面优先复用骨架，再在 page spec 中声明特例。

### 4.1 总览 / Dashboard 页面骨架

- 标题区组织：
  - 页面标题 + 时间范围/环境范围说明 + 1 个主操作（可选）。
- 操作区/工具栏：
  - 位于标题区下方或同层右侧，包含时间粒度切换、刷新、关键筛选。
- 内容区结构：
  - 以“摘要指标区 + 关键趋势区 + 风险与异常区”分段组织。
  - 常见为混合多列（宽屏多列，窄屏收敛为单列）。
- 适配模块类型：
  - KPI 指标卡、趋势图、健康状态列表、告警摘要、待处理任务。
- 应避免模式：
  - 把 dashboard 变成长列表堆叠。
  - 把所有过滤项塞入首屏，导致标题区失焦。

### 4.2 列表 / 管理页面骨架

- 标题区组织：
  - 页面标题 + 列表对象说明 + 主操作（创建、导入、批量动作入口）。
- 操作区/工具栏：
  - 标题下固定“筛选/搜索/批量操作/视图切换”层。
  - 批量操作仅在有选中对象时显性提升。
- 内容区结构：
  - 主体为表格或列表容器，可配可选统计条或二级信息面板。
  - 推荐单主栏 + 可选次级面板。
- 适配模块类型：
  - 实体管理、队列管理、权限条目、Key 列表、日志列表。
- 应避免模式：
  - 每个列表页采用不同工具栏顺序。
  - 在同一页面混用多种主列表交互模型（例如无限滚动与分页并行）。

### 4.3 详情 / 检查页面骨架

- 标题区组织：
  - 对象名称/标识 + 状态标签 + 主次操作（编辑、启停、危险动作入口）。
- 操作区/工具栏：
  - 对象级动作紧邻标题区；页面内次级动作归属到各区块头部。
- 内容区结构：
  - 推荐双栏或“主内容 + 次级上下文”混合布局。
  - 主栏承载核心配置与运行信息，侧栏承载状态摘要/关系对象/时间线。
- 适配模块类型：
  - Provider 详情、任务详情、用户详情、权限检查页。
- 应避免模式：
  - 详情页退化为无分组长表单。
  - 把所有高风险操作散落到内容底部难以发现。

### 4.4 设置 / 表单页面骨架

- 标题区组织：
  - 设置域标题 + 影响范围说明 + 变更风险提示（如涉及系统行为）。
- 操作区/工具栏：
  - 页面级操作简化为“保存/重置/测试连接”等，避免与字段级操作混淆。
- 内容区结构：
  - 推荐单栏分组或“主表单 + 辅助说明”双栏。
  - 长表单按主题分段，段内保持字段密度一致。
- 适配模块类型：
  - 系统设置、Provider 配置、通知配置、安全策略配置。
- 应避免模式：
  - 同一页面混用过多输入样式与对齐方式。
  - 在移动端保留桌面双栏导致字段可读性下降。

## 5. 区块规则

本节定义跨页面共享区块的层级关系、复用方式与稳定性要求。

### 5.1 页面标题与摘要区

- 层级：页面一级入口，始终位于页面最上层。
- 组成：标题、对象上下文、状态、主操作。
- 复用要求：结构顺序跨页面保持一致。
- 稳定性要求：不应在不同页面随意切换“标题在左/在上/居中”等模式。

### 5.2 筛选 / 搜索 / 批量操作工具栏

- 层级：位于标题区之后、主内容之前。
- 组成：筛选条件、检索入口、视图控制、批量动作。
- 复用要求：列表与管理页面优先复用统一工具栏模式。
- 稳定性要求：批量操作触发规则（无选中隐藏/禁用）跨页面一致。

### 5.3 表格区

- 层级：列表页主内容核心区块。
- 复用要求：表头、排序、分页、状态列位置遵循统一约定。
- 稳定性要求：不在不同页面频繁改变“密度、列对齐、行操作位”的基本模式。

### 5.4 卡片区

- 层级：用于摘要、分组状态或模块概览。
- 复用要求：卡片应承载明确语义，不作为任意信息容器。
- 稳定性要求：同类指标卡在同产品域保持一致的信息层级（标题、主值、辅助值、状态）。

### 5.5 详情信息区

- 层级：详情页主栏核心内容。
- 复用要求：按“基础信息、运行状态、关联对象、操作历史”组织可复用段落。
- 稳定性要求：避免每个详情页重新发明分组方式。

### 5.6 表单区

- 层级：设置页与编辑页主内容核心区。
- 复用要求：字段分组、说明文本、校验反馈、危险操作隔离模式统一。
- 稳定性要求：避免同类字段在不同页面使用不同标签对齐与错误提示位置。

### 5.7 底部 Action 区

- 层级：表单提交与页面保存的最终操作区。
- 复用要求：主动作、次动作、危险动作排序在同类页保持一致。
- 稳定性要求：长表单场景应保证提交动作可达，不得仅依赖页面顶部按钮。

### 5.8 空状态 / 错误状态区

- 层级：区块级或页面级反馈层。
- 复用要求：空状态应包含原因与下一步动作；错误状态应包含恢复路径。
- 稳定性要求：同类错误反馈的语气与结构保持一致，不做装饰化差异。

## 6. 响应式布局规则

本节为本规范核心，定义后台共享骨架在不同断点下的明确行为。

### 6.1 断点分层

- Desktop：宽屏工作区，适合多区并行查看与操作。
- Tablet：中等宽度设备，保留后台结构但降低并排密度。
- Mobile：窄屏设备，优先任务可达与信息可读，采用单列主流程。

注：实现层可按前端技术栈映射具体像素断点；本规范关注行为一致性，不绑定单一像素值。

### 6.2 Sidebar 响应式行为

- Desktop：
  - 默认展示侧栏，可折叠。
  - 折叠后不改变主内容信息架构，仅改变可视占比。
- Tablet：
  - 侧栏可默认收拢为窄态或隐藏态，由菜单按钮显式唤起。
  - 打开侧栏时应具备遮罩与关闭路径。
- Mobile：
  - 侧栏以抽屉为主，不常驻。
  - 导航跳转后自动关闭抽屉，避免遮挡内容。

### 6.3 Topbar / Page Header 响应式行为

- Desktop：标题、摘要、主操作可并排组织。
- Tablet：标题与操作可分两行；次级说明折叠或弱化显示。
- Mobile：
  - 标题区收敛为单列，主操作优先保留。
  - 次级操作进入更多菜单，不与主操作抢占首屏。

### 6.4 列表页与表格页在小屏下的处理

- Desktop：完整列模型 + 辅助列信息。
- Tablet：
  - 收敛低优先级列，优先保留身份列、状态列、关键动作。
  - 允许表格容器内部横向滚动，但页面整体不应横向滚动。
- Mobile：
  - 优先转为“列表卡片化”或“关键列 + 详情展开”模式。
  - 批量操作收敛为顶部动作入口或底部浮动入口，不覆盖行内主操作。

### 6.5 详情页双栏到单栏的转换

- Desktop：主栏 + 次级上下文栏并行。
- Tablet：
  - 次级栏可下沉到主栏之后，或以可折叠区块呈现。
- Mobile：
  - 统一转单栏，按“核心信息 -> 状态与诊断 -> 历史与关联”顺序堆叠。
  - 高风险操作固定在可达区域，不依赖横向布局位置。

### 6.6 设置页 / 表单页在窄屏下的堆叠规则

- Desktop：可采用分组单栏或主辅双栏。
- Tablet：优先单栏分组；说明性内容可折叠。
- Mobile：
  - 字段全量单列堆叠，标签与输入保持稳定顺序。
  - 底部 action 区固定可达或在段尾重复关键提交动作。

### 6.7 Dialog / Drawer / Side Panel 的窄屏优先行为

- Desktop：
  - 对象级编辑优先 Dialog，辅助浏览优先 Side Panel。
- Tablet：
  - 复杂交互优先 Drawer，减少小尺寸弹窗拥挤。
- Mobile：
  - 复杂编辑与多步骤操作优先全宽 Drawer 或全屏层。
  - Modal 仅用于短确认，不承载长表单主流程。

### 6.8 横向滚动约束

- 允许横向滚动：
  - 表格容器内部（列数受业务限制时）。
  - 代码片段/日志预格式化容器。
- 不允许横向滚动：
  - 页面根容器。
  - 标题区、主操作区、普通文本内容区。
- 约束原则：
  - 横向滚动必须是局部且可预期，不得把全页交互转化为横向浏览。

## 7. 与 Page Spec 的边界

- 页面级特殊布局、模块顺序、业务状态例外、断点特例，仍必须写入 `docs/pages/*.md`。
- 本文档只沉淀共享布局规则与响应式骨架，不定义单页业务细节。
- 禁止把单页临时需求直接上升为全局布局规范；仅当多页面复用且稳定后再提升。

## 8. 与实现及评审的关系

### 8.1 前端实现基线

- 前端实现应将本规范视为后台共享布局基线。
- 若 page spec 未声明特例，默认按本规范实现 Shell、骨架与响应式行为。
- 组件实现可随技术栈演进，但布局语义与层级规则不应漂移。

### 8.2 UI Review 基线

- UI 评审应同时对照：
  - `DESIGN.md` 的系统级视觉/组件规则
  - 本规范的跨页面布局骨架与响应式规则
  - 对应 `docs/pages/*.md` 的页面级要求
- 评审重点应包含：
  - 跨页面布局一致性
  - 响应式行为是否符合骨架规则
  - 是否出现未声明的页面级布局偏离
