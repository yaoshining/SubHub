# Page Spec

## Metadata

- **Page**: Access Control
- **Route / Entry Point**: `access-control.html` / `/access-control`
- **Status**: Draft
- **Last Updated**: 2026-05-24
- **Related Feature Specs**: `specs/001-mvp-admin-console/spec.md`

## Goal

为管理员提供一个集中治理**角色模板、作用域矩阵、敏感操作审批与最近权限变更**的权限策略页，用来回答“谁可以看、谁可以改、哪些动作必须审批、最近权限策略刚刚发生了什么变化”。

## Audience / Scenario

- **Primary user**: 管理员 / owner
- **Primary scenario**: 审查系统内预设角色的可见范围与可执行动作，决定是否收紧审批护栏并发布权限策略
- **Frequency / importance**: 低频但高风险治理页；不属于日常运营主路径，但直接影响后台高风险操作边界

## Modules / Sections

1. **页头与策略动作区**: 说明权限治理定位，并提供“收紧审批”“发布策略”等高风险动作入口。
2. **治理摘要卡区**: 展示角色数量、受保护动作数量、审批规则数量和审计告警数量，帮助管理员先判断当前策略强度。
3. **角色预设列表区**: 展示系统预设角色，并允许切换查看不同角色的作用域与限制。
4. **作用域矩阵区**: 以动作 × 角色的方式展示 allow / review / block 的治理结果，帮助管理员理解差异。
5. **选中角色详情区**: 展示当前角色的作用域数量、审批人数要求、适用说明与能力边界。
6. **审批护栏区**: 展示与当前角色相关的敏感动作护栏，例如双人确认、二次认证、自动进入审计流。
7. **最近权限变更审计流**: 展示最近策略变更与权限收紧动作，帮助管理员确认发布后的影响。

## Key States

- **Default state**: 默认展示标准护栏模式下的摘要卡、角色预设、作用域矩阵、选中角色详情和最近审计流。
- **Loading state**: 优先展示页头与摘要卡骨架，再加载角色列表、矩阵和审计流；右侧详情区应保留结构骨架，避免页面跳空。
- **Empty state**: 当暂无可展示的权限策略数据时，页面必须明确说明“当前未加载到角色策略模板”，并提示管理员检查初始化或回退到默认策略，而不是显示空白双栏。
- **Error state**: 当角色策略、矩阵或审计流加载失败时，页面必须指出失败对象，并保留已成功加载的区块与重试入口。
- **Permission / access state**: 仅管理员 / owner 可访问与发布策略；无权限用户不得看到可操作的发布入口，也不能通过隐藏交互绕过限制。
- **Guardrail-strict state**: 启用“收紧审批”后，摘要卡中的 guarded actions / approval rules / audit alerts 数值与护栏文案必须同步切换到严格模式。
- **Policy-published state**: 发布策略后，页面必须提供明确成功反馈，并让最近权限变更审计流立即出现对应发布记录。

## Content Hierarchy

- **Primary information**: 当前角色体系如何划分、哪些动作允许 / 需审批 / 被阻断、当前策略是否已收紧。
- **Secondary information**: 角色作用域数量、审批要求、角色说明、护栏细节、最近变更记录。
- **Tertiary information**: 时间戳、客户端角色说明、审计补充文案、策略模式解释。
- **Primary actions**: 收紧审批、发布策略、切换查看角色。
- **Secondary actions**: 浏览矩阵、查看审计流、理解各角色适用范围。

## Interaction Rules

- 本页承接的是**权限策略治理**，不是成员分配页；具体谁属于哪个角色仍由 `users` 页面承接，不得在本页混入成员列表或邀请流程。
- 角色为系统预设模板；当前版本只允许查看、切换和发布预设策略，不支持新增自定义角色或自由定义任意 RBAC 结构。
- 切换角色时，作用域矩阵高亮、角色详情与审批护栏内容必须同步更新，避免用户在双栏结构中失去上下文。
- 作用域矩阵中的 allow / review / block 必须同时通过文案与状态样式表达，不能只靠颜色区分。
- “收紧审批”属于模式切换，不是立即发布；切换后应先更新页面内派生状态，再由“发布策略”完成正式落地。
- “发布策略”是高风险写入动作，必须具备明确确认与结果反馈；发布成功后应写入最近权限变更审计流。
- 审批护栏应围绕当前选中角色展示，不得扩展成全量系统设置长表单。
- 审计流在本页承担“最近变更可见性”职责，不替代完整审计日志中心或导出能力。

## Page-Specific Design Rules

- **Relevant global rules from `DESIGN.md`**: `2. 产品定位与界面原则`、`6.2 页面布局模式`、`6.3 密度原则`、`7.1 按钮`、`7.2 卡片与面板`、`7.4 表格与列表`、`7.5 状态标签、Badge、Chip`、`7.6 Inline Callout 与 Notice`、`7.8 导航`、`7.9 组件系统基线`、`8. 状态与反馈规则`、`9. 数据展示与技术信息呈现`
- **Allowed overrides**: 作为治理页，本页允许在双栏结构中提升矩阵信息密度，并允许用 `review` 这类中间态强调审批语义；但仍必须保持主次层级稳定，不把整页做成彩色权限墙。
- **Forbidden deviations**: 不得把本页扩展成成员管理页、组织架构页或通用系统设置页；不得支持无边界的自定义角色创建；不得把高风险发布动作弱化成普通链接或隐藏在次级菜单中；不得让矩阵状态只靠颜色表达。

## Component Patterns (shadcn/ui)

本页面组件实现必须优先映射到以下 shadcn/ui 组件结构，不得随意自造基础控件。

### 页头与策略动作区

- 页面标题与说明：`h1` + 辅助说明文本。
- “收紧审批”：`Button variant="outline"`，可配合受控开关语义或按钮态切换。
- “发布策略”：`Button variant="default"`，必要时配合 `AlertDialog` 做高风险确认。

### 治理摘要卡区

- 外层容器：多列 `Card` 网格。
- 单张摘要卡：`Card` + `CardHeader` / `CardContent`。
- 告警类摘要必须使用 `Badge` 或状态说明文本，不能只放一个数字。

### 角色预设列表区

- 外层容器：`Card`。
- 角色列表：使用结构化列表或按钮组；每个角色项应具备选中态，可映射为 `Button` 变体或列表行选中样式。
- 角色说明、作用域数等补充信息使用 `Badge` 与次级文本组合呈现。

### 作用域矩阵区

- 外层容器：`Card` with `CardHeader` 和 `CardContent`。
- 矩阵本体优先使用 `Table`。
- allow / review / block 状态统一映射为 `Badge` 或受控状态单元格，不自造颜色体系。

### 选中角色详情区

- 外层容器：`Card`。
- 角色名称、角色类型与限制摘要使用标题 + `Badge` 组。
- 作用域数量、审批要求、适用说明使用结构化信息行、`Separator` 或轻量子分区组织。

### 审批护栏区

- 外层容器：`Card`。
- 单条护栏可使用结构化列表、`Alert` 或带状态标记的信息块。
- 涉及双确认、二次认证、自动审计等风险要求时，优先使用 `Badge` + 说明文本，不做复杂可视化。

### 最近权限变更审计流

- 外层容器：`Card`。
- 变更流使用结构化垂直列表，辅以 `Separator` 或轻量时间轴样式。
- 时间戳、角色名、动作对象等技术性信息应使用清晰的主次文本层级，必要时对关键标识使用等宽样式。

## Data / Dependencies

- **Data sources**: 角色预设列表、角色作用域数量、动作权限矩阵、审批护栏规则、策略模式（standard / strict）、最近权限变更记录
- **Derived UI state**: 当前选中角色、当前护栏模式、矩阵状态映射、发布后新增的审计记录、摘要卡统计数
- **External dependencies**: 无
- **Cross-page dependencies**: `docs/pages/users.md`、`docs/pages/settings.md`、`docs/pages/api-keys.md`、`docs/pages/provider-detail.md`

## Responsive Behavior (Page-Specific)

> 共享响应式骨架规则参见 `docs/layouts/admin-layout.md §6`；以下为本页特有例外与优先级约定。

### Tablet

- 左侧角色预设列表与右侧矩阵 + 详情区可调整为垂直堆叠，角色列表在上、矩阵在下。
- 治理摘要卡在 Tablet 下保持多列横向排布；若卡片数 > 3，可折叠为 2 列网格。
- "发布策略"等高风险动作按钮必须在首屏可达，不得被折叠至溢出菜单。

### Mobile

- 统一转单列：治理摘要卡 → 角色预设列表 → 选中角色详情（含矩阵视图）→ 护栏区 → 审计流。
- 权限矩阵（动作 × 角色）在 Mobile 下优先采用**横向滚动**策略，保持矩阵完整性，不折叠角色列。
- 若矩阵列数超过 3 列，需确保第一列（动作名称）固定不滚动，便于用户对照行信息横扫角色列。
- "发布策略"高风险动作在 Mobile 下必须保留 `AlertDialog` 二次确认，不允许因屏幕窄而降级为普通 `Button`。

## Notes

- 本页是 `users` 页中角色分配能力的上游治理页：`users` 负责“人”，本页负责“角色规则与审批边界”。
- 本页也是 `settings` 页中“复杂权限模型”分流后的承接页：`settings` 只负责告知去向，不在其内重复权限矩阵与审计治理。
- 当前原型表达的是**预设角色治理**而不是通用权限构建器；若未来引入自定义角色、租户级隔离或完整审计中心，应另开 feature，而不是直接塞进本页。
- “API Client” 角色属于非人类角色模板，页面必须清楚表达其为 scope 受限的调用身份，避免与后台成员角色混淆。
