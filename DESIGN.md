# SubHub 设计文档

## 1. 文档目的

本文档定义 SubHub 管理控制台原型当前的产品设计方向。其目标是让产品、设计与前端实现围绕同一套信息架构、运行模型与页面职责达成一致。

本文档反映的是本项目当前原型文件，而不是一个假设中的未来系统。

## 2. 产品概述

SubHub 是一个自托管字幕网关，面向需要以下能力的团队或运维人员：

- 聚合多个字幕提供方
- 缓存字幕资源以减少上游重复拉取
- 以人工上传作为运维兜底手段
- 保护提供方凭证与 API 配额，避免过度消耗
- 对媒体工作流与内部工具暴露受控的外部 API

管理控制台不是通用仪表盘。它的职责是让运维事实可见：

- 哪些提供方健康
- 哪个令牌池正在承压
- 哪些字幕已缓存、存在冲突或被人工修复
- 哪些外部客户端正在消耗 API 容量
- 哪些用户被允许修改高风险设置

## 3. 设计目标

- 在统一系统中清晰呈现提供方健康度、缓存价值与故障切换行为。
- 将令牌池、API Key、权限与人工干预视为一等产品对象。
- 保持 UI 可落地实现：真实控件、真实状态变化、清晰页面边界。
- 支持响应式 Web，而不是把移动布局压缩成拥挤的桌面布局。
- 保持产品控制台气质，避免漂移为营销页或通用后台模板。

## 4. 主要用户角色

### 运维管理员

负责系统健康、队列恢复、提供方调优、缓存治理与告警处理。

### 集成负责人

负责媒体服务器连接、Webhook 状态、API 客户端与下游同步行为。

### 安全或平台管理员

负责角色、权限范围、密钥生命周期与敏感操作防护。

### 支持或内容运营人员

负责上传审核、冲突处理与任务级人工干预。

## 5. 产品原则

- 提供方管理是核心工作流，不是设置页子模块。
- 令牌池是共享资源，必须清晰暴露轮换、冷却与配额压力。
- 缓存是产品资产，不是文件系统细节。
- 人工上传与自动抓取属于同一运维图谱。
- 围绕 Key、提供方编辑与全局操作，访问控制必须明确。
- 移动端响应式应保持同一产品外壳与任务流程，导航折叠为抽屉。

## 6. 视觉与交互系统

### 主题

- 默认主题：深色
- 备选主题：浅色
- 主题状态在页面间持久化

### 组件方向

当前 UI 方向有意接近 `tailwindcss + shadcn/ui` 的控制台风格：

- 深色中性画布与克制边框
- 使用分层表面而非重阴影
- 弱化元信息、强化内容层级
- 使用分段控件进行模式切换
- ghost / secondary / primary 的按钮层级
- 以内联状态芯片反馈系统状态

### 外壳行为

- 桌面端使用常驻左侧边栏。
- 在较窄宽度下，边栏变为由菜单按钮打开的隐藏抽屉。
- 移动端抽屉包含遮罩、关闭动作，并在导航后自动关闭。
- 不同断点下导航目标结构必须保持一致。

## 7. 规范化信息架构

当前规范化管理控制台包含 16 个页面：

1. `dashboard.html` — 运维总览
2. `task-queue.html` — 任务列表与重试管理
3. `subtitle-library.html` — 缓存清单与预览
4. `subtitle-conflicts.html` — 版本冲突治理
5. `uploads.html` — 人工上传审核与替换流程
6. `providers.html` — 提供方列表、健康度与令牌池态势
7. `provider-detail.html` — 单提供方配置与令牌控制
8. `failover-history.html` — 故障切换历史与恢复轨迹
9. `media-integrations.html` — Jellyfin、Plex、Emby 与 Webhook 状态
10. `alerts.html` — 事件规则、通知通道与静默窗口
11. `task-detail.html` — 单任务执行轨迹与人工兜底
12. `api-analytics.html` — 请求量、429、P95 与客户端压力
13. `api-keys.html` — API Key 生命周期管理
14. `users.html` — 成员、邀请、会话与账号风险
15. `access-control.html` — 角色、权限范围、审批防护与审计
16. `settings.html` — 缓存、存储、Webhook 与系统全局态势

该 IA 是导航的事实来源。新增页面只应在其代表稳定产品对象或稳定工作流时引入，而不是为了临时设计便利。

## 8. 页面职责

### 8.1 运维核心

#### Dashboard

目的：

- 传达当前系统态势
- 暴露当前需要处理的事项
- 将操作员引导到正确的深层页面

必须展示：

- 全局健康信号
- 队列压力
- 数据源健康快照
- 人工干预数量
- 覆盖缺口
- 下一步运维指引

#### Task Queue

目的：

- 按状态浏览任务
- 重试或检查失败任务
- 识别积压集中区域

必须承担：

- 队列分诊的控制台
- 通往任务详情的桥接页

#### Task Detail

目的：

- 端到端解释单个任务
- 展示实际走过的提供方路径
- 说明何时需要人工兜底

必须包含：

- 拉取路径或重试轨迹
- 当前状态
- 干预动作
- 失败证据

### 8.2 字幕资产治理

#### Subtitle Library

目的：

- 管理缓存字幕资产
- 区分热资产与冷候选资产
- 提供预览、来源与使用价值信息

必须展示：

- 资产列表
- 命中次数与最近使用时间
- 来源归因
- 版本与状态标识
- 直达冲突治理的入口

#### Subtitle Conflicts

目的：

- 治理同一媒体与语言下存在多个竞争字幕版本的场景

必须展示：

- 受影响资产
- 当前对外服务版本
- 候选版本
- 版本胜出或淘汰原因
- 解决冲突或固定优选版本的操作路径

#### Uploads

目的：

- 审核人工提交的字幕
- 接受、拒绝或替换字幕版本

必须邻接：

- 字幕库
- 任务恢复
- 冲突治理

### 8.3 提供方运维

#### Providers

目的：

- 一眼比较提供方价值与风险
- 展示每个提供方的令牌池压力
- 引导进入配置详情

必须展示：

- 提供方健康度
- 成功率或退化状态
- 令牌池规模
- 冷却与配额风险
- 人工源与自动源角色

#### Provider Detail

目的：

- 配置单个提供方
- 管理凭证、权重、冷却与故障回退行为

必须包含：

- 令牌池表格
- 运行时策略控制
- 提供方权重
- 冷却窗口
- 回退目标
- 保存动作

该页面闭合了提供方管理回路，是“在哪里填写 Key 并调优提供方行为”这一问题的主要答案。

#### Failover History

目的：

- 解释系统如何随时间恢复
- 展示故障切换是保持自动化还是升级为人工干预

必须展示：

- 事件列表
- 选中事件详情
- 数据源不稳定模式
- 恢复率

### 8.4 集成与告警

#### Media Integrations

目的：

- 管理下游媒体服务器连接
- 发现外部同步问题如何放大字幕失败

必须展示：

- 已连接服务器
- Webhook 失败
- 同步延迟
- 集成维度状态

#### Alerts

目的：

- 配置哪些事件需要被关注
- 定义通道、静默窗口与严重等级响应

必须展示：

- 激活规则
- 通知通道
- 静默窗口
- 最近发送
- 选中规则详情

### 8.5 外部 API 与访问控制

#### API Analytics

目的：

- 展示谁在调用 API
- 展示外部客户端施加的系统压力
- 区分内部 API 保护与上游提供方限制

必须展示：

- 请求量
- 成功率
- 429 事件
- 延迟
- 热点端点
- 客户端压力

#### API Keys

目的：

- 将 API Key 视为受管理的产品资源

必须包含：

- Key 清单
- 创建流程
- 权限范围
- 环境隔离
- 轮换
- 暂停
- 配额状态

#### Users

目的：

- 管理成员、邀请、会话与高风险账号

#### Access Control

目的：

- 定义角色、权限范围、审批规则与审计态势

必须保护：

- 提供方编辑
- Key 可见性
- 全局变更
- 敏感运维动作

#### Settings

目的：

- 管理横切的平台策略

包含：

- 缓存 TTL 态势
- 存储行为
- Webhook 默认策略
- 全局功能开关

## 9. 关键产品工作流

### 9.1 提供方故障与令牌故障切换

目标流程：

1. 操作员在 Dashboard 或 Providers 看到退化信号。
2. 操作员打开 Providers 检查健康度与令牌池压力。
3. 操作员打开 Provider Detail 调整权重、并发、冷却或回退目标。
4. 操作员查看 Failover History 验证恢复行为。

设计含义：

- 令牌轮换与冷却不能隐藏在通用设置表单中
- 提供方详情必须暴露可执行控制项

### 9.2 缓存价值与版本治理

目标流程：

1. 操作员打开 Subtitle Library 检查缓存价值。
2. 若资产存在多个版本，则进入 Subtitle Conflicts。
3. 若无可用版本，则进入 Uploads 或 Task Detail 进行恢复。

设计含义：

- 字幕库与冲突页职责分离
- 字幕库回答“我们有什么？”
- 冲突页回答“应该服务哪个版本？”

### 9.3 外部 API 治理

目标流程：

1. 操作员在 API Analytics 看到配额或请求突增。
2. 识别对应客户端或热点端点。
3. 进入 API Keys 进行轮换、暂停或签发新 Key。
4. 若涉及更大范围权限变更，则进入 Access Control。

设计含义：

- API 统计与 API Key 生命周期需要分屏承载
- Key 需要具备状态、范围与环境语义

### 9.4 用户与角色治理

目标流程：

1. 管理员检查用户、邀请与会话异常。
2. 管理员打开 Access Control 验证角色权限与受保护动作。
3. 敏感动作需要明确审批或审计覆盖。

## 10. 响应式约束

本项目目标是响应式 Web，而非原生移动应用。

要求：

- 桌面端保持侧边栏可见
- 小视口切换为隐藏式导航抽屉
- 控件保持可触控
- 顶栏动作可以换行，但不能塌缩到不可用密度
- 任何页面都不应依赖仅悬停可见的交互

至少应在以下断点检查新 UI：

- 360px
- 390px 到 430px
- 768px 到 834px
- 1024px 到 1180px
- 1280px 及以上

## 11. 内容规则

- 使用真实运维语言，不用抽象仪表盘套话。
- 避免无意义的虚构指标。
- 优先使用能解释系统行为、风险或修复路径的表述。
- 若数值未知，应展示诚实状态，而不是伪精确数据。

## 12. 当前原型边界

当前原型处于“线框到结构化产品”的过渡阶段，已包含：

- 真实页面边界
- 真实导航
- 深色与浅色主题
- 移动端抽屉导航
- 基础交互钩子

当前尚不宣称：

- 可直接生产的后端绑定
- 最终文案
- 最终版组件库实现
- 每行表格都具备深度下钻流程

## 13. 已知扩展方向

高价值后续增强：

- 面向查询级调试的提供方检索测试台
- 按媒体与语种的覆盖率分析
- 按 Key 与客户端下钻的 API 分析
- 带审计历史的用户详情抽屉
- 可主动选择对外服务字幕版本的冲突处理动作

## 14. 实施说明

- `index.html` 是启动页与总览入口。
- `screens/` 包含面向用户的产品页面。
- `assets/nav.js` 定义规范化的 16 页面导航模型。
- 共享样式与外壳行为应保持集中，除非某页面明确需要覆写。

新增页面时，必须同时更新：

- `assets/nav.js`
- `index.html`
- 本 `DESIGN.md`

以保持原型、启动页与设计文档同步。
