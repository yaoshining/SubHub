# SubHub 设计文档

## 1. 文档目的

本文档定义 SubHub 管理后台原型的当前设计方向，作为产品、设计与前端实现的共同基线。

它描述的是当前项目中已经落地的原型结构与交互约束，不是未来某个假设版本的愿景稿。

---

## 2. 产品定位

SubHub 是一个面向团队和运营者的自托管字幕网关，核心职责包括：

- 聚合多个字幕 Provider
- 缓存字幕资源，减少重复上游请求
- 处理人工上传与人工兜底
- 管理 Provider 凭据与 Token 池，降低单个 Token 被打爆的风险
- 向媒体工作流或内部系统暴露受控 API

这个后台不是通用 BI 看板，也不是单纯的“系统设置页集合”。它要把运营真相直接展示出来：

- 当前哪些 Provider 健康，哪些在抖动
- 哪些 Token 池接近额度上限，哪些处于冷却
- 当前缓存里到底有什么字幕资产
- 哪些任务失败后触发了回退或人工补救
- 哪些外部客户端正在消耗 API 容量
- 谁拥有修改高风险设置的权限

---

## 3. 设计目标

- 让 Provider 健康度、缓存价值、回退行为在同一套控制台里可读、可管、可追溯。
- 把 Token 池、API Key、权限、人工上传都做成一等产品对象，而不是埋在说明文字里。
- 保持“可实现”的原型标准：真实控件、明确状态、清晰页面边界，而不是静态截图。
- 支持响应式 Web 使用，在窄屏下维持同一套产品结构，而不是把桌面页面硬缩小。
- 保持专业控制台气质，避免滑回通用后台模板或营销页语气。

---

## 4. 核心用户角色

### 4.1 运营管理员

负责系统健康、任务队列、Provider 调优、缓存治理与告警响应。

### 4.2 集成负责人

负责 Jellyfin / Plex / Emby 等媒体服务器接入、Webhook、同步状态与 API 客户端管理。

### 4.3 安全或平台管理员

负责角色权限、Scope、审批护栏、API Key 生命周期与高风险动作控制。

### 4.4 支持或内容运营

负责人工上传、任务修复、冲突处理与资源纠错。

---

## 5. 产品原则

- Provider 管理是核心工作流，不是系统设置的子页。
- Token 池是共享资源，必须清楚展示轮换、冷却、失败切换和额度压力。
- 缓存是产品资产，不是文件系统细节。
- 自动抓取与人工补传属于同一条运营链路，不能割裂。
- 权限控制必须围绕 Key、Provider 配置、上传与全局设置显式展开。
- 响应式不能改变信息架构，只改变导航与布局方式。

---

## 6. 视觉与交互系统

### 6.1 主题

- 默认主题：深色
- 备选主题：浅色
- 主题状态跨页面持久化
- 登录页与后台页必须共享同一套主题逻辑，不能出现独立入口风格漂移

### 6.2 组件方向

当前 UI 风格刻意贴近 `tailwindcss + shadcn/ui` 的后台控制台语气：

- 深色中性画布，边框克制
- 通过层级与表面区分信息，不靠重阴影堆叠
- 元信息弱化，关键数字和标题强化
- 使用 segmented control 做模式切换
- 按钮层级清晰区分 primary / secondary / ghost
- 通过状态 chip、badge、inline callout 表达系统反馈

### 6.3 页面外壳

- 桌面端使用固定左侧导航
- 窄屏下导航收拢为隐藏式抽屉
- 抽屉必须带菜单按钮、遮罩、关闭动作、Esc 关闭
- 移动端抽屉点击导航后自动关闭
- 桌面与移动端目的地结构保持一致

### 6.4 响应式原则

SubHub 当前目标是响应式 Web，而不是原生 App：

- 宽屏：完整侧栏 + 多列信息面板
- 中屏：保留控制台结构，减少并排密度
- 窄屏：导航改抽屉，内容改单列或少列
- 不允许把桌面卡片机械压缩成手机截图式布局

## 7. 规范化信息架构

当前后台的规范化信息架构包含 17 个 screen：

1. `login.html`：登录入口、工作区选择、2FA / SSO
2. `dashboard.html`：运营总览
3. `task-queue.html`：任务队列
4. `subtitle-library.html`：字幕库 / 缓存浏览
5. `subtitle-conflicts.html`：字幕版本冲突治理
6. `uploads.html`：上传管理
7. `providers.html`：Provider 列表与健康态
8. `provider-detail.html`：单个 Provider 配置详情
9. `failover-history.html`：回退历史
10. `media-integrations.html`：媒体服务器集成
11. `alerts.html`：告警规则与通知
12. `task-detail.html`：任务详情
13. `api-analytics.html`：API 调用统计
14. `api-keys.html`：API Key 管理
15. `users.html`：用户管理
16. `access-control.html`：权限策略
17. `settings.html`：系统设置

这是当前原型的权威导航结构。新增页面只能在它代表稳定产品对象或独立工作流时引入，不能为了展示方便随意扩 screen。

## 8. 页面职责定义

### 8.1 认证入口

#### 登录页 `login.html`

用途：

- 在进入后台前确认操作者身份
- 选择目标工作区或环境
- 承载密码登录、SSO、审计只读模式与 2FA 流程

必须表达：

- 这是一个高权限后台，不是普通 SaaS 落地页
- 生产环境和演示环境有边界
- 高风险动作默认受二次验证保护

---

### 8.2 运营核心

#### 运营总览 `dashboard.html`

用途：

- 告诉运营者“现在系统整体怎么样”
- 快速暴露最值得处理的问题
- 把人导向正确的下钻页面

必须包含：

- 系统健康概览
- 队列压力
- Provider 健康摘要
- 缓存命中或覆盖面信号
- 人工介入数量
- 明确的下一步动作入口

#### 任务队列 `task-queue.html`

用途：

- 浏览任务列表
- 按状态筛选任务
- 对失败任务进行批量重试或逐个进入详情

必须包含：

- pending / retrying / failed / done 等状态筛选
- 批量动作
- 错误聚类或失败摘要
- 到任务详情的明确入口

#### 任务详情 `task-detail.html`

用途：

- 解释单个任务的执行链路
- 展示失败点、重试轨迹与人工兜底动作

必须包含：

- 当前状态
- 抓取与重试轨迹
- 使用过的 Provider 路径
- 上传备用字幕或人工修复入口

---

### 8.3 字幕资产治理

#### 字幕库 `subtitle-library.html`

用途：

- 浏览当前缓存中的字幕资产
- 按影片、语言、来源、命中情况搜索
- 预览文件、管理冷热资产

必须包含：

- 资产列表
- 命中次数 / 最近访问时间
- 来源归属
- 状态与版本信号
- 到冲突治理页的入口

#### 字幕版本冲突 `subtitle-conflicts.html`

用途：

- 管理同一媒体与语言下的多个字幕版本冲突
- 明确当前“主服务版本”与候选版本之间的关系

必须包含：

- 冲突资产信息
- 当前主服务版本
- 候选版本列表
- 裁决依据
- 手动指定或固定首选版本的入口

#### 上传管理 `uploads.html`

用途：

- 集中管理人工上传记录
- 对上传结果做审核、替换、删除和关联任务追踪

必须包含：

- 上传历史
- 文件状态
- 关联任务
- 删除 / 替换 / 接受动作

---

### 8.4 Provider 运营

#### Provider 管理 `providers.html`

用途：

- 横向比较 Provider 的价值与风险
- 暴露 Token 池压力
- 从列表视图进入配置详情

必须包含：

- Provider 健康状态
- 成功率或抖动状态
- Token 池规模与压力
- 冷却 / 配额预警
- 进入详情页的明确 CTA

#### Provider 配置详情 `provider-detail.html`

用途：

- 管理单个 Provider 的运行策略
- 配置凭据、权重、并发、冷却与回退目标

必须包含：

- Token 池明细
- 轮换与冷却策略
- 并发限制
- 权重调整
- fallback 目标
- 保存动作

这是 Provider 管理闭环的关键页面，用来回答“API Key 在哪填、优先级在哪调、冷却策略在哪设”。

#### 回退历史 `failover-history.html`

用途：

- 观察系统过去如何从主源失败切换到备用源
- 判断自动回退是否健康，还是已经频繁升级为人工介入

必须包含：

- 回退事件列表
- 事件详情
- 失败源与接手源
- 自动恢复 / 人工升级结果

---

### 8.5 集成与告警

#### 媒体集成 `media-integrations.html`

用途：

- 管理 Jellyfin / Plex / Emby 等接入端点
- 测试连接、查看同步状态与错误

必须包含：

- 端点列表
- 鉴权状态
- 最近同步状态
- 测试连接动作
- 异常提示

#### 告警规则 `alerts.html`

用途：

- 配置系统何时告警、发到哪里、何时静默

必须包含：

- 告警类型
- 通知渠道
- 静默窗口
- 升级规则或阈值策略

---

### 8.6 API 能力治理

#### API 调用统计 `api-analytics.html`

用途：

- 看清 API 消耗结构与压力来源
- 识别热点端点、高压客户端、429 与延迟异常

必须包含：

- 请求量
- 成功率
- 429 / 限流情况
- P95 延迟
- 热点端点
- 高消耗客户端

#### API Key 管理 `api-keys.html`

用途：

- 管理外部 API 凭据的生命周期
- 支持生成、轮换、停用、筛选和使用情况回看

必须包含：

- Key 列表
- 环境 / Scope / 配额
- 最近使用
- 生成与轮换入口
- 停用或撤销动作

---

### 8.7 用户与权限治理

#### 用户管理 `users.html`

用途：

- 管理成员、邀请、会话与风险账号

必须包含：

- 成员列表
- 邀请入口
- 会话状态
- 暂停 / 恢复动作
- 风险提示

#### 权限策略 `access-control.html`

用途：

- 定义角色、Scope、审批强度与敏感动作护栏

必须包含：

- 角色矩阵
- Scope 模板
- 审批或双人确认策略
- 审计事件流

---

### 8.8 全局系统配置

#### 系统设置 `settings.html`

用途：

- 管理全局缓存、存储、Webhook 与系统级策略

边界：

- 只放真正的系统全局配置
- 不吞并 Provider、权限、API Key 这类已经独立成面的高频对象

## 9. 关键产品闭环

### 9.1 Provider 闭环

`providers` 列表发现风险 → 进入 `provider-detail` 调整 Token 池、权重、冷却 → 到 `failover-history` 观察调整后回退是否减少。

### 9.2 任务修复闭环

`task-queue` 发现失败任务 → 进入 `task-detail` 看抓取链路 → 必要时进入 `uploads` 做人工补传 → 回到任务状态确认恢复。

### 9.3 缓存治理闭环

在 `subtitle-library` 浏览资产与命中 → 发现异常版本或重复 → 进入 `subtitle-conflicts` 做冲突裁决 → 必要时回到 `uploads` 替换版本。

### 9.4 API 治理闭环

在 `api-analytics` 识别高压端点或 429 → 到 `api-keys` 看具体调用方与 Key 生命周期 → 必要时回到 `access-control` 调整 Scope 或审批策略。

### 9.5 权限治理闭环

在 `users` 管成员与会话 → 在 `access-control` 定角色与护栏 → 确保高风险动作被正确约束。

## 10. Token 池与失败切换设计原则

SubHub 里的免费或低配额 Provider Token 通常存在明显限制，因此后台必须把 Token 池管理做清楚。

设计上应遵守：

- 一个 Provider 下可以有多个 Token
- Token 状态至少区分：active、cooldown、quota-risk、disabled
- 失败切换必须可见，不能只在后端默默发生
- 单个 Token 不应因流量倾斜被持续打爆
- 429、认证失败、配额耗尽要进入不同的处理路径

界面上至少需要体现：

- Token 池概览
- 当前使用中的 Token
- 失败后切换到哪个 Token
- 冷却窗口
- 近 24 小时 failover 次数
- 配额压力与高风险 Token 标记

---

## 11. 导航与结构规则

- 导航是全局真实导航，不是演示用目录。
- 每一页都必须能跳到其它主要页面，不能出现空 `<nav>`。
- `index.html` 是 launcher / 总览入口，不应承担所有产品内容。
- Screen 数量、导航项、文档描述必须同步；任何一处新增或删减都要同步更新。
- 不允许在最终产品页面里暴露“设计器开关”“viewport 选择器”“demo controls”。

---

## 12. 实现约束

- 每个用户可感知的独立工作面保持单独 HTML 文件。
- 复杂页面可以使用共享 `assets/*.js` 与 `assets/styles.css`，但页面职责不能因此混淆。
- 所有高频动作至少要有基础交互反馈，不能只有静态按钮。
- 登录页、后台页、launcher 的主题逻辑保持一致。
- 响应式导航切换必须通过真实菜单抽屉，而不是简单把侧栏挪到顶部。

## 14. 后续优先级建议

下一阶段更值得补深度，而不是继续盲目扩 screen 数量。优先建议：

1. `Provider 搜索测试台`
   用于直接输入影片名测试单个 Provider 的返回，帮助排查接入质量。

2. `健康诊断 / 部署检查`
   查看数据库、存储空间、依赖服务连通性、Webhook 状态等运维健康面。

3. `用户详情抽屉 / 审计下钻`
   强化权限变更与敏感动作追踪。

4. `API analytics drill-down`
   按 Key / client / endpoint 深入查看消耗结构。

5. `冲突治理增强`
   支持固定主版本、查看差异预览、批量处理重复冲突。

---

## 15. 文档维护规则

- 新增页面时，先更新本文档中的信息架构和页面职责，再更新 launcher 与导航。
- 删除页面时，必须同步移除导航、入口卡片与相关闭环描述。
- 改动主题或响应式行为时，优先更新“视觉与交互系统”与“实现约束”两节。
- 本文档优先反映“当前真实原型”，而不是口头设想。
