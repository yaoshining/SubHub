### 设计约束

- 实现 MUST 遵循 `DESIGN.md` 中"Provider 管理是核心工作流""令牌池是共享资源""API Key 是受管理产品资源"的规则。
- 控制台后台 Shell MUST 遵循 `docs/layouts/admin-layout.md §3` 定义的 Sidebar / Topbar / Main Content / Secondary Panel 结构。
- 响应式行为 MUST 遵循 `docs/layouts/admin-layout.md §6` 定义的断点骨架规则（Desktop ≥1280px / Tablet 768–1279px / Mobile <768px）；各 page spec 中声明的页面级响应式特例具有更高优先级。
- 侧边栏导航项与页面标题 MUST 使用中文（仪表盘 / 服务商 / API 密钥 / 用户 / 设置），遵循 `DESIGN.md §7.8`。
- 图标 MUST 使用 Lucide 图标体系（前端使用 `lucide-react`），图标命名遵循 `DESIGN.md §7.8` 及 `docs/layouts/admin-layout.md §5.8.1` 中的语义映射；不得混用其他图标库。
- 所有基础界面元素（按钮、输入框、表格、弹层、标签、卡片等）MUST 优先复用 `shadcn/ui` 组件（遵循 `DESIGN.md §7.9`）；各 page spec 的 `Component Patterns (shadcn/ui)` 章节为强制实现基线，仅在 page spec 明确说明理由时才允许自定义基础组件。
- 空状态展示 MUST 遵循 `docs/layouts/admin-layout.md §5.8.1` 定义的卡片结构（Icon + Title + Body + CTA），并使用 page spec 中指定的 Lucide 图标（服务商: `cloud-off`，API 密钥: `key-round`，用户: `users`）。
- 本功能的首个可交付版本 MUST 聚焦于单维护者可运营的最小闭环，而非一次覆盖全部治理能力。
- 登录、Provider 配置、API Key 管理与统一出口服务 MUST 使用一致的运维语言，不引入营销式表述。
- Users 页 MUST 聚焦后台访问生命周期的最小治理：成员可见、轻量邀请、暂停/恢复与基础会话处置；不得扩展为完整身份治理、复杂风控系统或权限矩阵功能。
- Settings 页 MUST 保持"系统状态确认 + 配置分流 + 后续能力边界"的低频边界页定位，不得演变为通用系统配置中心。

## 用户场景与测试 *(mandatory)*

### 用户故事 1 - 首次开通控制台 (Priority: P1)

作为系统维护者，我希望能建立并使用一个管理员账号登录控制台，以便完成系统首轮开通并进入后续配置流程。

**优先级原因**: 没有受控登录入口，就没有安全的配置面，也无法开展 Provider 与对外接口管理。

**独立测试**: 通过建立管理员账号、完成登录并进入控制台首页，可独立验证系统具备最基础的受控运营入口。

**验收场景**:

1. **Given** 系统尚未完成首轮开通，**When** 维护者完成首个管理员身份建立，**Then** 系统创建可用的管理员访问入口并引导进入控制台。
2. **Given** 管理员身份已建立，**When** 管理员使用有效凭据登录，**Then** 系统展示受保护的控制台首页。
3. **Given** 未登录用户访问控制台页面，**When** 其尝试进入 Provider 或 API Key 管理页面，**Then** 系统要求先完成身份验证。

---

### 用户故事 2 - 配置 Provider 与凭据池 (Priority: P2)

作为系统维护者，我希望在控制台中配置一个字幕提供方并维护多个可切换凭据，以便系统在额度紧张或单个凭据失效时仍可持续服务。

**优先级原因**: 这是首个 MVP 的核心运营价值，直接决定系统能否稳定地连接上游字幕来源。

**独立测试**: 通过新增一个上游 Provider、录入多个凭据、查看凭据状态并调整启停状态，可独立验证 Provider 管理闭环。

**验收场景**:

1. **Given** 管理员已登录控制台，**When** 管理员配置一个启用状态的字幕提供方，**Then** 系统保存该提供方的基本信息与可运营状态。
2. **Given** 某提供方存在多个可用凭据，**When** 其中一个凭据达到限制或被管理员停用，**Then** 系统将其移出可用池并保留其他活跃凭据继续参与服务。
3. **Given** 管理员查看某提供方详情，**When** 其进入凭据池视图，**Then** 系统展示每个凭据的状态、可用性与最近异常信息。

---

### 用户故事 3 - 对外提供统一字幕查询与下载 (Priority: P3)

作为接入方应用负责人，我希望获得一个受控的统一字幕服务入口，以便通过一个稳定接口完成字幕查询与下载，而不需要直接接入上游字幕站点。

**优先级原因**: 这是 MVP 的最终用户价值出口，决定 SubHub 是否真正承担“统一字幕网关”的角色。

**独立测试**: 通过创建一个可用的调用方 Key，并用该凭据完成一次查询和一次下载，可独立验证对外服务闭环。

**验收场景**:

1. **Given** 管理员已为某调用方创建可用 Key，**When** 该调用方使用有效凭据发起字幕查询，**Then** 系统返回统一结果集或明确的无结果反馈。
2. **Given** 查询结果中存在可用字幕项，**When** 调用方发起下载请求，**Then** 系统返回对应字幕内容或明确失败原因。
3. **Given** 调用方使用已停用或无效 Key，**When** 其访问统一字幕服务，**Then** 系统拒绝请求并返回可识别的授权失败结果。

---

### 用户故事 4 - 管理后台成员与基础会话风险 (Priority: P4)

作为系统维护者，我希望能查看谁可以登录后台、邀请新的维护成员、暂停或恢复成员，并处置需要关注的后台会话，以便在单维护者或小团队场景下维持最小访问安全。

**优先级原因**: 登录能力进入 MVP 后，后台访问权本身成为高风险对象；当前版本不需要完整权限体系或风控平台，但需要最小成员生命周期与基础会话处置闭环。

**独立测试**: 通过创建成员邀请、查看待接受邀请、暂停/恢复成员、处置需要关注的后台会话，可独立验证 Users 页承担的是最小后台访问治理，而不是完整组织管理。

**验收场景**:

1. **Given** 管理员已登录控制台，**When** 管理员创建成员邀请，**Then** 系统生成待接受邀请并在成员/邀请列表中展示状态。
2. **Given** 某成员不应继续访问后台，**When** 管理员暂停该成员，**Then** 系统阻止该成员继续登录或执行受保护后台管理动作。
3. **Given** 某成员已被暂停且需要恢复访问，**When** 管理员恢复该成员，**Then** 系统允许其重新进入可用成员状态，并在成员列表中反映最新状态。
4. **Given** 系统存在需要管理员关注的后台会话，**When** 管理员完成基础会话处置，**Then** 系统更新会话状态、风险摘要和成员详情，并保留明确的处置结果。
5. **Given** 当前登录管理员查看自身成员详情，**When** 其尝试执行暂停动作，**Then** 系统拒绝该操作，并在页面与接口中明确返回“当前登录管理员不能暂停自己”的原因，而不是静默失败或普通网络错误。
6. **Given** 系统当前只剩最后一个 active admin，**When** 任意管理员尝试暂停该账号，**Then** 系统拒绝该操作，并在页面与接口中明确返回“最后一个 active admin 不可被暂停”的原因。

---

### 用户故事 5 - 确认系统状态并找到正确配置入口 (Priority: P5)

作为系统维护者，我希望在系统设置页快速确认实例是否已具备基础服务条件，并明确知道不同配置应该去哪个页面完成，以避免把低频信息页误当成总配置中心。

**优先级原因**: 这不是 MVP 主闭环的核心交易路径，但它决定管理员能否正确理解当前版本的职责边界，减少错误导航和错误配置预期。

**独立测试**: 通过进入 Settings 页、读取系统状态、识别未就绪原因并跳转到正确治理页，可独立验证该页承担的是“确认与分流”职责而非深度编辑。

**验收场景**:

1. **Given** 管理员已登录控制台，**When** 其进入系统设置页，**Then** 系统展示环境、系统版本、首个管理员状态和服务就绪度等基础状态摘要。
2. **Given** 管理员想调整 Provider 策略、API Key 或成员治理，**When** 其查看系统设置页的配置分流区，**Then** 系统明确指向对应页面而不是在当前页提供深度编辑入口。
3. **Given** 当前系统尚未满足对外服务条件，**When** 管理员查看系统设置页，**Then** 页面明确指出未就绪信号或缺失条件，并引导前往对应治理页处理。

---

### 边界场景

- 当系统尚未配置任何启用中的 Provider 时，查询请求必须返回明确的“服务未就绪”状态，而不是模糊失败。
- 当某 Provider 下所有凭据均不可用时，系统必须阻止继续分配请求到失效凭据。
- 当管理员停用某个调用方 Key 时，新的外部请求必须不再被接受。
- 当成员被暂停时，其既有后台会话必须失效或不再允许执行受保护后台动作。
- 当当前登录管理员尝试暂停自己时，系统必须明确拒绝，并返回可识别的业务原因，而不是把该失败表现为普通请求异常。
- 当目标成员是系统最后一个 active admin 时，系统必须明确拒绝暂停，并保留至少一个可用管理员入口。
- 当成员邀请尚未接受、已过期或已撤销时，Users 页必须清楚展示邀请状态，而不是把其混同为活跃成员。
- 当基础会话处置失败时，页面必须指出失败对象，并保留当前成员与会话上下文。
- 当上游返回无字幕结果时，系统必须区分“无结果”与“请求失败”。
- 当管理员尝试删除仍被使用中的凭据或调用方 Key 时，系统必须要求先停用或替换，避免造成无感中断。
- 当系统设置页无法获取部署读数或核查摘要时，页面必须保留已知可用信息，并明确指出失败对象，而不是把整页降为空白。
- 当管理员误把系统设置页当成全局配置中心时，页面必须通过范围说明和配置分流明确指出深配置应前往的治理页。

## 需求 *(mandatory)*

### 功能需求

- **FR-001**: 系统 MUST 提供受保护的管理控制台访问入口，并允许建立至少一个管理员身份用于登录。
- **FR-002**: 系统 MUST 在未认证状态下阻止访问受保护的控制台页面与高风险管理动作。
- **FR-003**: 管理员 MUST 能够查看控制台首页中的首轮运营状态，包括是否已登录、是否已配置 Provider、是否已创建调用方 Key。
- **FR-004**: 系统 MUST 允许管理员新增、启用、停用和查看至少一个字幕提供方配置。
- **FR-005**: 系统 MUST 允许管理员为单个 Provider 维护多个可独立启停的凭据。
- **FR-006**: 系统 MUST 在 Provider 凭据不可用、超额或被手动停用时，自动将新的上游请求切换到同一 Provider 的其他活跃凭据。
- **FR-007**: 系统 MUST 为每个 Provider 凭据保留可运营状态信息，至少包括可用状态、最近异常信息与最近使用情况。
- **FR-008**: 系统 MUST 允许管理员创建、查看、停用和轮换对外调用方 Key。
- **FR-009**: 每个调用方 Key MUST 绑定明确的调用方标识和当前状态，以支持后续运营追踪。
- **FR-010**: 系统 MUST 对外提供统一的字幕查询能力，使外部应用无需直接理解上游 Provider 差异。
- **FR-011**: 系统 MUST 对外提供统一的字幕下载能力，并在下载不可用时返回明确原因。
- **FR-012**: 系统 MUST 校验外部调用方 Key 的有效性，并拒绝已停用、无效或未授权的请求。
- **FR-013**: 系统 MUST 在管理控制台中清晰区分上游 Provider 凭据与下游调用方 Key，避免运营对象混淆。
- **FR-014**: 系统 MUST 记录关键管理动作的结果，包括 Provider 启停、凭据停用、调用方 Key 停用与轮换结果。
- **FR-015**: 系统 MUST 允许管理员在不影响其他已启用凭据的前提下，单独隔离异常凭据。
- **FR-016**: 系统 MUST 让管理员能够判断系统当前是否具备对外服务能力，包括至少一个可用 Provider 和至少一个可用调用方 Key。
- **FR-017**: 系统 MUST 提供一个只读的系统设置总览页，用于展示环境、系统版本、管理员初始化状态和服务就绪度。
- **FR-018**: 系统设置页 MUST 明确区分“当前可在本版本完成的配置”和“应前往其他页面完成的深配置”，避免管理员把该页当成总配置中心。
- **FR-019**: 当系统未具备基础服务条件时，系统设置页 MUST 明确暴露未就绪信号或缺失条件，并指向对应治理页。
- **FR-020**: 系统 MUST 提供 Users 页用于展示后台成员、待接受邀请、成员状态和基础会话风险摘要。
- **FR-021**: 管理员 MUST 能够创建成员邀请；MVP 中邀请仅允许使用预设角色和接入范围，不允许在 Users 页定义角色策略或权限矩阵。
- **FR-022**: 管理员 MUST 能够暂停和恢复后台成员；被暂停成员不得继续登录或执行受保护后台管理动作。
- **FR-023**: 系统 MUST 展示需要管理员关注的后台会话摘要，并允许管理员执行基础会话处置；该能力仅面向当前后台安全维护需要，不等于完整风控平台或身份治理系统。
- **FR-024**: 系统 MUST 记录成员邀请、暂停、恢复和基础会话处置的关键动作结果。
- **FR-025**: 系统 MUST 禁止当前登录管理员暂停自己；Users 页面与管理端接口都必须返回明确拒绝原因，不得静默失败，也不得把该限制表现为普通网络错误。
- **FR-026**: 系统 MUST 禁止暂停系统最后一个 active admin；Users 页面与管理端接口都必须返回明确拒绝原因，并保护至少一个可用管理员入口始终存在。

### 非功能需求 *(mandatory)*

- **NFR-001 (代码质量)**: Feature MUST 定义格式化、静态检查与代码规范门禁，并作为进入实现前的显式约束。
- **NFR-002 (测试)**: Feature MUST 定义覆盖登录访问控制、Provider 凭据切换、调用方 Key 授权、查询与下载主流程的单元测试与集成/契约测试。
- **NFR-003 (UX 一致性)**: Feature MUST 对管理员与外部调用方分别提供一致、可识别的成功、失败、未就绪与未授权反馈。
- **NFR-004 (性能)**: Feature MUST 以“可用于日常调用”为目标，要求大多数合法查询请求在可接受等待时间内返回结果或明确状态，且凭据切换不造成长时间请求挂起。
- **NFR-005 (设计保真)**: Feature MUST 遵循 `DESIGN.md` 及各 page spec 中的信息分层与控制台交互风格；`docs/layouts/admin-layout.md` 作为共享布局规范，与 `DESIGN.md` 同级约束。
- **NFR-006 (并行隔离)**: Feature MUST 绑定当前 active feature 目录 `specs/001-mvp-admin-console`，并保持 worktree 仅跟踪该功能。
- **NFR-007 (Issue 同步范围)**: 后续 issue 同步 MUST 仅面向 `specs/001-mvp-admin-console`，不得混入其他功能任务。
- **NFR-008 (响应式)**: 所有后台页面 MUST 在 Mobile / Tablet / Desktop 三个断点下遵循 `docs/layouts/admin-layout.md §6` 的响应式骨架行为；页面级特例（Provider Detail 次级栏下沉、API Keys 双栏堆叠、Users 批量操作收敛等）遵循对应 page spec 的 Responsive Behavior 章节。
- **NFR-009 (组件实现)**: 基础 UI 元素实现 MUST 优先映射到 `shadcn/ui` 组件；仅当 page spec 明确说明现有组件无法承载时，才允许引入自定义基础组件，并须在 page spec 中记录理由。
- **NFR-010 (图标一致性)**: 前端图标实现 MUST 使用 `lucide-react`；图标命名须与 page spec 和设计稿保持一致，不得跨页面使用语义不一致的图标，不得混用其他图标库。

### 关键实体 *(如功能涉及数据请填写)*

- **管理员账号**: 可访问管理控制台的受控身份，负责登录、配置 Provider、管理调用方 Key 和执行高风险操作。
- **后台成员**: 可登录管理控制台的受控身份，具有成员状态、预设角色、最近活动和会话摘要；MVP 不在此实体上承载完整 RBAC 或权限矩阵。
- **成员邀请**: 管理员创建的待接受访问入口，具有接收标识、预设角色、接入范围、状态和过期信息。
- **后台会话**: 成员登录后的访问会话，具有成员归属、最近活动、基础风险状态和是否可处置的状态；MVP 仅支持基础会话处置，不做高级风险分析或风控策略系统。
- **字幕提供方**: 一个可被启用或停用的上游字幕来源，具有名称、运行状态和是否可服务的运营属性。
- **Provider 凭据**: 绑定到某个字幕提供方的单个可切换凭据，具有可用状态、最近异常信息、最近使用情况与是否参与调度的属性。
- **调用方 Key**: 分配给外部应用的访问凭据，具有调用方标识、状态、轮换历史与是否可继续访问统一字幕服务的属性。
- **字幕查询请求**: 外部应用发起的查询动作，包含目标媒体识别信息、请求状态与结果集合。
- **字幕下载请求**: 外部应用基于可用结果项发起的下载动作，包含授权状态、目标字幕项与交付结果。

## 成功标准 *(mandatory)*

### 可度量结果

- **SC-001**: 90% 的首次维护者可在 10 分钟内完成管理员登录、配置一个可用 Provider，并创建至少一个调用方 Key。
- **SC-002**: 95% 的合法查询请求可在 5 秒内返回统一结果集或明确的无结果状态。
- **SC-003**: 当同一 Provider 下存在至少两个活跃凭据时，单个凭据失效后，100% 的新请求都不会继续分配到该失效凭据。
- **SC-004**: 100% 的无效或已停用调用方 Key 请求都返回明确的授权失败结果，而不会误返回业务成功响应。
- **SC-005**: 管理员可在 2 分钟内完成单个异常凭据或调用方 Key 的停用操作，并能立即从控制台看到状态变化。
- **SC-006**: 90% 的已登录管理员可在 1 分钟内通过系统设置页判断当前实例是否具备基础服务条件，并找到正确的下钻治理页面。
- **SC-007**: 管理员可在 2 分钟内完成单个成员邀请、暂停/恢复成员或基础会话处置，并能立即在 Users 页看到对应状态变化。

## 假设

- 首个版本仅面向单实例、自托管、单维护者或小团队使用场景。
- 首个版本只要求接入一个明确的上游字幕来源，OpenSubtitles 作为默认首发 Provider。
- 首个版本不包含公开注册、复杂角色体系、审批流或多租户能力。
- 首个版本的统一对外能力只覆盖“查询字幕”和“下载字幕”，不覆盖上传、审核、冲突治理与高级分析。
- 管理员需要区分“上游 Provider 凭据”与“下游调用方 Key”，并能分别运营这两类对象。
- Users 页在首个版本中仅覆盖后台访问生命周期的最小治理，包括成员邀请、暂停/恢复和基础会话处置；不包含权限矩阵、审批流、完整 RBAC、审计导出、高级风险分析或风控策略系统。
- 系统设置页在首个版本中仅承担信息确认、部署读数和配置分流职责，不承接全局可编辑策略表单。
- 若后续需要更多 Provider、权限模型、统计分析和告警能力，应在后续 feature 中逐步扩展，而不是并入当前 MVP。
