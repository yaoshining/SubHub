## 用户场景与测试 *(mandatory)*

### 用户故事 1 - 首次开通控制台 (Priority: P1)

作为系统维护者，我希望能建立并使用一个管理员账号登录控制台，以便完成系统首轮开通并进入后续配置流程。

**优先级原因**: 没有受控登录入口，就没有安全的配置面，也无法开展 Provider 与对外接口管理。

**独立测试**: 通过建立管理员账号、完成登录并进入控制台首页，可独立验证系统具备最基础的受控运营入口。

**验收场景**:

1. **Given** 系统尚未完成首轮开通，**When** 维护者完成首个管理员身份建立，**Then** 系统创建可用的管理员访问入口并引导进入控制台。
2. **Given** 管理员身份已建立，**When** 管理员使用有效凭据登录，**Then** 系统展示受保护的控制台首页。
3. **Given** 未登录用户访问控制台页面，**When** 其尝试进入 Provider 或 API Key 管理页面，**Then** 系统要求先完成身份验证。

---

### 用户故事 2 - 配置 Provider 与凭据池 (Priority: P2)

作为系统维护者，我希望在控制台中配置一个字幕提供方并维护多个可切换凭据，以便系统在额度紧张或单个凭据失效时仍可持续服务。

**优先级原因**: 这是首个 MVP 的核心运营价值，直接决定系统能否稳定地连接上游字幕来源。

**独立测试**: 通过新增一个上游 Provider、录入多个凭据、查看凭据状态并调整启停状态，可独立验证 Provider 管理闭环。

**验收场景**:

1. **Given** 管理员已登录控制台，**When** 管理员配置一个启用状态的字幕提供方，**Then** 系统保存该提供方的基本信息与可运营状态。
2. **Given** 某提供方存在多个可用凭据，**When** 其中一个凭据达到限制或被管理员停用，**Then** 系统将其移出可用池并保留其他活跃凭据继续参与服务。
3. **Given** 管理员查看某提供方详情，**When** 其进入凭据池视图，**Then** 系统展示每个凭据的状态、可用性与最近异常信息。

---

### 用户故事 3 - 对外提供统一字幕查询与下载 (Priority: P3)

作为接入方应用负责人，我希望获得一个受控的统一字幕服务入口，以便通过一个稳定接口完成字幕查询与下载，而不需要直接接入上游字幕站点。

**优先级原因**: 这是 MVP 的最终用户价值出口，决定 SubHub 是否真正承担“统一字幕网关”的角色。

**独立测试**: 通过创建一个可用的调用方 Key，并用该凭据完成一次查询和一次下载，可独立验证对外服务闭环。

**验收场景**:

1. **Given** 管理员已为某调用方创建可用 Key，**When** 该调用方使用有效凭据发起字幕查询，**Then** 系统返回统一结果集或明确的无结果反馈。
2. **Given** 查询结果中存在可用字幕项，**When** 调用方发起下载请求，**Then** 系统返回对应字幕内容或明确失败原因。
3. **Given** 调用方使用已停用或无效 Key，**When** 其访问统一字幕服务，**Then** 系统拒绝请求并返回可识别的授权失败结果。

---

### 边界场景

- 当系统尚未配置任何启用中的 Provider 时，查询请求必须返回明确的“服务未就绪”状态，而不是模糊失败。
- 当某 Provider 下所有凭据均不可用时，系统必须阻止继续分配请求到失效凭据。
- 当管理员停用某个调用方 Key 时，新的外部请求必须不再被接受。
- 当上游返回无字幕结果时，系统必须区分“无结果”与“请求失败”。
- 当管理员尝试删除仍被使用中的凭据或调用方 Key 时，系统必须要求先停用或替换，避免造成无感中断。

## 需求 *(mandatory)*

### 功能需求

- **FR-001**: 系统 MUST 提供受保护的管理控制台访问入口，并允许建立至少一个管理员身份用于登录。
- **FR-002**: 系统 MUST 在未认证状态下阻止访问受保护的控制台页面与高风险管理动作。
- **FR-003**: 管理员 MUST 能够查看控制台首页中的首轮运营状态，包括是否已登录、是否已配置 Provider、是否已创建调用方 Key。
- **FR-004**: 系统 MUST 允许管理员新增、启用、停用和查看至少一个字幕提供方配置。
- **FR-005**: 系统 MUST 允许管理员为单个 Provider 维护多个可独立启停的凭据。
- **FR-006**: 系统 MUST 在 Provider 凭据不可用、超额或被手动停用时，自动将新的上游请求切换到同一 Provider 的其他活跃凭据。
- **FR-007**: 系统 MUST 为每个 Provider 凭据保留可运营状态信息，至少包括可用状态、最近异常信息与最近使用情况。
- **FR-008**: 系统 MUST 允许管理员创建、查看、停用和轮换对外调用方 Key。
- **FR-009**: 每个调用方 Key MUST 绑定明确的调用方标识和当前状态，以支持后续运营追踪。
- **FR-010**: 系统 MUST 对外提供统一的字幕查询能力，使外部应用无需直接理解上游 Provider 差异。
- **FR-011**: 系统 MUST 对外提供统一的字幕下载能力，并在下载不可用时返回明确原因。
- **FR-012**: 系统 MUST 校验外部调用方 Key 的有效性，并拒绝已停用、无效或未授权的请求。
- **FR-013**: 系统 MUST 在管理控制台中清晰区分上游 Provider 凭据与下游调用方 Key，避免运营对象混淆。
- **FR-014**: 系统 MUST 记录关键管理动作的结果，包括 Provider 启停、凭据停用、调用方 Key 停用与轮换结果。
- **FR-015**: 系统 MUST 允许管理员在不影响其他已启用凭据的前提下，单独隔离异常凭据。
- **FR-016**: 系统 MUST 让管理员能够判断系统当前是否具备对外服务能力，包括至少一个可用 Provider 和至少一个可用调用方 Key。

### 非功能需求 *(mandatory)*

- **NFR-001 (代码质量)**: Feature MUST 定义格式化、静态检查与代码规范门禁，并作为进入实现前的显式约束。
- **NFR-002 (测试)**: Feature MUST 定义覆盖登录访问控制、Provider 凭据切换、调用方 Key 授权、查询与下载主流程的单元测试与集成/契约测试。
- **NFR-003 (UX 一致性)**: Feature MUST 对管理员与外部调用方分别提供一致、可识别的成功、失败、未就绪与未授权反馈。
- **NFR-004 (性能)**: Feature MUST 以“可用于日常调用”为目标，要求大多数合法查询请求在可接受等待时间内返回结果或明确状态，且凭据切换不造成长时间请求挂起。
- **NFR-005 (设计保真)**: Feature MUST 遵循 `DESIGN.md` 中 Dashboard、Providers、Provider Detail、API Keys、Users 的信息分层与控制台交互风格。
- **NFR-006 (并行隔离)**: Feature MUST 绑定当前 active feature 目录 `specs/001-mvp-admin-console`，并保持 worktree 仅跟踪该功能。
- **NFR-007 (Issue 同步范围)**: 后续 issue 同步 MUST 仅面向 `specs/001-mvp-admin-console`，不得混入其他功能任务。

### 关键实体 *(如功能涉及数据请填写)*

- **管理员账号**: 可访问管理控制台的受控身份，负责登录、配置 Provider、管理调用方 Key 和执行高风险操作。
- **字幕提供方**: 一个可被启用或停用的上游字幕来源，具有名称、运行状态和是否可服务的运营属性。
- **Provider 凭据**: 绑定到某个字幕提供方的单个可切换凭据，具有可用状态、最近异常信息、最近使用情况与是否参与调度的属性。
- **调用方 Key**: 分配给外部应用的访问凭据，具有调用方标识、状态、轮换历史与是否可继续访问统一字幕服务的属性。
- **字幕查询请求**: 外部应用发起的查询动作，包含目标媒体识别信息、请求状态与结果集合。
- **字幕下载请求**: 外部应用基于可用结果项发起的下载动作，包含授权状态、目标字幕项与交付结果。

## 成功标准 *(mandatory)*

### 可度量结果

- **SC-001**: 90% 的首次维护者可在 10 分钟内完成管理员登录、配置一个可用 Provider，并创建至少一个调用方 Key。
- **SC-002**: 95% 的合法查询请求可在 5 秒内返回统一结果集或明确的无结果状态。
- **SC-003**: 当同一 Provider 下存在至少两个活跃凭据时，单个凭据失效后，100% 的新请求都不会继续分配到该失效凭据。
- **SC-004**: 100% 的无效或已停用调用方 Key 请求都返回明确的授权失败结果，而不会误返回业务成功响应。
- **SC-005**: 管理员可在 2 分钟内完成单个异常凭据或调用方 Key 的停用操作，并能立即从控制台看到状态变化。

## 假设

- 首个版本仅面向单实例、自托管、单维护者或小团队使用场景。
- 首个版本只要求接入一个明确的上游字幕来源，OpenSubtitles 作为默认首发 Provider。
- 首个版本不包含公开注册、复杂角色体系、审批流或多租户能力。
- 首个版本的统一对外能力只覆盖“查询字幕”和“下载字幕”，不覆盖上传、审核、冲突治理与高级分析。
- 管理员需要区分“上游 Provider 凭据”与“下游调用方 Key”，并能分别运营这两类对象。
- 若后续需要更多 Provider、权限模型、统计分析和告警能力，应在后续 feature 中逐步扩展，而不是并入当前 MVP。
