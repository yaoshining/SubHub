## 项目简介

SubHub 用于在应用与多个字幕来源之间建立一层统一的字幕服务。

它的目标不是绑定某一个字幕站点，而是通过可插拔的 Provider 适配层，把不同字幕来源统一成一致的搜索、下载、缓存和上传能力，方便自部署、二次开发和后续扩展。

你可以把它理解为：

- 一个统一的字幕 API 网关
- 一个可缓存字幕文件和元数据的中间层
- 一个支持手动上传字幕的私有字幕服务
- 一个适合接入 Jellyfin、Plex、Emby 或自定义影视应用的后端基础设施

## 设计背景

很多字幕服务都会存在以下问题：

- API 调用频率受限
- 接口不稳定，甚至没有官方 API
- 不同字幕源返回格式不一致
- 某些来源访问受地区限制
- 缺少本地缓存能力
- 缺少上传、管理和审核能力
- 应用侧需要重复对接多个字幕站点

SubHub 希望通过统一接口和本地缓存机制，降低上游依赖带来的不稳定性，让应用只需要接入一次。

## 核心特性

- 聚合多个字幕来源，统一对外提供 API
- 支持字幕搜索、下载、缓存和管理
- 支持手动上传字幕
- 支持将不同来源的数据标准化
- 支持来源失效时的自动回退
- 支持私有化部署
- 适合作为二次开发基础项目

## 适用场景

- 为 Jellyfin、Plex、Emby 或自研播放器提供统一字幕接口
- 对第三方字幕服务进行缓存，减少重复请求
- 在公共字幕源之外补充自有字幕文件
- 同时接入国内外多个字幕来源
- 构建内部字幕管理平台或字幕 API 服务

## 计划支持的字幕来源

### 官方或较稳定的 API 来源

- OpenSubtitles
- 其他提供正式 API 的字幕平台

### 社区或第三方来源

- ASSRT
- Shooter
- Xunlei
- SubtitleBest
- 其他可通过适配器接入的字幕站点

### 本地来源

- 手动上传字幕
- 本地缓存字幕
- 自有字幕库

## 功能规划

- 统一的字幕搜索接口
- 统一的字幕下载接口
- 字幕文件缓存
- 字幕元数据缓存
- 手动上传字幕
- 字幕记录管理
- Provider 适配器机制
- 字幕来源优先级和回退策略
- 用户认证和权限控制
- 管理后台
- 与媒体服务器的集成能力

## 架构设计

SubHub 计划采用模块化架构：

### 1. API 层

负责对外暴露统一接口，例如：

- 搜索字幕
- 下载字幕
- 上传字幕
- 查询缓存
- 管理字幕记录

### 2. Provider 适配层

负责对接不同字幕来源，并将外部返回结果转换为统一格式。

每个来源都应该以独立 Provider 的方式接入，避免系统对单一字幕源产生强耦合。

### 3. 缓存层

负责缓存：

- 字幕文件
- 搜索结果
- 字幕元数据
- Provider 映射关系

缓存的目标是减少对上游站点的重复请求，并提升整体可用性。

### 4. 存储层

负责存储：

- 上传的字幕文件
- 标准化后的字幕记录
- 影片与字幕的关联关系
- 字幕来源和下载历史

### 5. 管理层

负责：

- Provider 配置
- 上传管理
- 字幕审核
- 缓存管理
- 访问控制

## 设计原则

- 面向多来源，而不是绑定单一字幕站
- 优先支持自托管部署
- 保持 Provider 插件化、可替换
- 对外接口保持稳定
- 方便扩展、方便二次开发
- 尽量降低上游变化对系统整体的影响

## 开发路线图

- 定义统一的 Provider 接口
- 接入 OpenSubtitles
- 实现本地缓存机制
- 实现字幕上传接口
- 实现标准化的字幕数据模型
- 增加搜索和下载历史
- 增加认证和权限控制
- 增加管理后台
- 增加更多国内字幕源适配器
- 增加与媒体服务器的集成能力

## 非目标

以下内容不在本项目的直接目标范围内：

- 在未确认条款前，公开再分发第三方字幕内容
- 绑定某一个具体播放器或媒体服务器
- 依赖单一字幕来源构建系统核心能力

## 合规说明

本项目定位为可自托管的字幕聚合与管理服务。

在接入任何第三方字幕来源前，请务必自行确认以下内容：

- API 调用限制
- 服务条款
- 缓存与再分发政策
- 版权要求
- 本地法律法规合规性

部分字幕来源可能并未提供公开、稳定或官方支持的 API。此类来源通常需要通过页面抓取、非官方接口或特定规则进行接入，维护成本和合规风险会更高。
